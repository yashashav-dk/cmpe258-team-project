Autonomous Bug-Squashing Agent

Planner → Executor → Critic loop for end-to-end Python bug repair

Abstract. LLM agent that locates and fixes Python bugs autonomously - read code, propose patch, run pytest, iterate until green. Evaluated on 52 synthetic cases and 3 injected defects in pallets/click v8.3.2 across 4 LLMs. Best result: 57% synth pass (Qwen 2.5 72B) and 33% OSS pass (Gemini 3 Pro), with 3.1–3.4× token reduction and 2.5–2.7× latency improvement over a Gemini CLI baseline at identical resolution outcomes.

Tools available to Planner

  • read_file(path, start, end) - targeted reads, not full file
  • edit_file(path, line_range, new_block) - atomic replace, AST-validated
  • run_bash(cmd) - pytest oracle, sandboxed scope

Key invariants

  • Per-case Memory - fresh state, no cross-case leakage
  • Atomic patch - write .tmp → ast.parse() → os.replace()
  • Path-traversal guard - realpath scope check before write
  • Golden never fed - used only for post-hoc evaluation

7. What Drives Performance

Model reasoning → Pass rate

Stronger reasoning models fix more bugs, independent of agent scaffolding. Gemini 3 Pro maps test traceback to mutation in 2 steps. Gemini 3 Flash, same scaffold, burns 10 retries on the same case.

On synth: Qwen 2.5 72B is 65% Easy → 22% Medium. Drop mirrors reasoning depth required, not architecture.

Agent design → Efficiency

  • Targeted tool calls - read only relevant function, not full file
  • Dead-end memory - skip patches already proven to fail
  • Structured retry context - compact diff, not full traceback resend
  • Planner persona - primes minimal JSON patch plans

Outcomes identical to baseline. Cost: 3.1–3.4× less input, 2.5–2.7× less time.

8. Conclusion & Future Work

Finding 1 - Model quality determines pass rate. Gap between models dwarfs any agent-level optimization. No retry logic compensates for insufficient reasoning depth on medium/hard.

Finding 2 - Agent design determines efficiency. 3.1–3.4× input-token reduction, 2.5–2.7× latency improvement, zero loss in resolution quality.

Finding 3 - Synthetic ≠ OSS difficulty. 57% synth vs ≤33% OSS reveals large-file localization and multi-module context as key unsolved challenges.

Future directions

  • AST-guided patch targeting - eliminate line-offset localization errors
  • Longer retry budgets with curriculum difficulty scaling
  • Cross-file retrieval (RAG over repo) for architectural bugs
  • Hard-tier synthetic cases to bridge gap with OSS complexity
